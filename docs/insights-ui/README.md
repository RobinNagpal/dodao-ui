# Insights-UI Knowledge

Topical reference docs for the Insights-UI (KoalaGains) app — patterns, prompts, runbooks, analysis methodology, plus the active task lists that drive ongoing work.

## Subfolders

- **[etf-analysis/](etf-analysis/)** — How the ETF analysis pipeline works end-to-end: report generation and the ETF Scenarios system (winner / loser model + admin/automation paths). See [etf-analysis/README.md](etf-analysis/README.md).
- **[stock-analysis/](stock-analysis/)** — Runbooks for adding new stocks (`yarn stocks:add`) and triggering stock report generation (`yarn stocks:trigger`). See [stock-analysis/README.md](stock-analysis/README.md).
- **[etf-prompts/](etf-prompts/)** — Source-of-truth prompt text for each ETF analysis category (past returns, cost & efficiency, risk, future outlook, intro/strategy, competition) plus the prompt-finalization approach. See [etf-prompts/README.md](etf-prompts/README.md).
- **[scenario-prompts/](scenario-prompts/)** — Reusable prompt templates for stock & ETF scenario authoring. Currently: `detailed-analysis.md` for generating the optional `detailedAnalysis` long-form section.
- **[stock-prompts/](stock-prompts/)** — Source-of-truth prompt text for stock analysis report types (currently `management-team`). The prompts themselves live in the `prompt_versions` DB table; the files here are the human-readable, version-controlled copy used for review and seeding. See [stock-prompts/README.md](stock-prompts/README.md).
- **[etf-prompt-improvement/](etf-prompt-improvement/)** — Methodology + runbook for the prompt-tuning loop (`etf-verification-loop.md`, `run-prompt-analysis.md`) plus iterative prompt-review notes, factor-set reviews, and per-ETF audits. See [etf-prompt-improvement/README.md](etf-prompt-improvement/README.md).
- **[tariffs/](tariffs/)** — Reference docs for the tariffs subsystem: pipeline, data structures, UI components, S3 storage, admin flow, and the use-case catalog that drives feature prioritization.
- **[tasks/](tasks/)** — Active KoalaGains task lists (open + closed work, per surface): ETFs, stocks, tariffs, scenarios, prompt tuning, plus open questions and per-page checklists.

## Top-level files

- **[ui-api-routes.md](ui-api-routes.md)** — Inventory of every `GET` / `POST` / `PUT` / `DELETE` API route backing the stocks, stock/ETF scenarios, ETFs, prompts, and prompt-invocation surfaces, with the auth wrapper each handler uses. Documents the rule that every stocks/ETFs/prompts/prompt-invocation mutation uses `withAdminOrToken` (admin JWT **or** automation token), and the explicit exceptions: machine-to-machine **callbacks**, **cron/batch** triggers, and user-owned **favourites** (which stay on `withLoggedInUser`). Read before adding or reviewing a mutating endpoint in these areas.
- **[ui-leaf-component-system.md](ui-leaf-component-system.md)** — The styling architecture for insights-ui: a small leaf layer (`src/components/ui/**`) owns ALL Tailwind, while high-level components (admin/stock/ETF/scenario pages) carry zero `className` and compose leaves. Covers the leaf inventory, the `cva` + `cn` authoring pattern, semantic color tokens, the ESLint `no-restricted-syntax` guardrail (warn → per-directory ratchet to error), and the step-by-step process for adding a leaf or migrating a high-level component.
- **[automated-report-generation.md](automated-report-generation.md)** — Five CLI scripts (`stocks:prompt`, `stocks:save`, `stocks:list-oldest`, `etfs:prompt`, `etfs:save`) that let Claude drive the full 7-ETF / 8-stock report-generation loop, plus a "refresh the N stocks with the oldest reports" batch flow: setup, report order, per-type response JSON shapes, run-time gotchas (JSON-escape quotes, partial MOR data on first prompt, `final-summary` must be last), and the end-to-end copy-paste skeleton.
- **[scenario-authoring.md](scenario-authoring.md)** — How to draft a new stock or ETF market scenario: scratch-file workflow under `/tmp/scenarios/{stocks,etfs}/<slug>.md`, the required content template (dates, magnitude, per-industry impact numbers, asymmetric winners / losers), per-stock bullet syntax, the outlook phrasing the parser keys off, and the editorial "five winners, five losers" convention.
- **[stock-page-caching.md](stock-page-caching.md)** — Reference for the ISR / Data Cache topology under `/stocks/[exchange]/[ticker]` and its subpages: per-route tag map, every code path that invalidates a `ticker_*` tag, and the structural reason ISR writes still exceed reads after PRs #1472/#1473 (each rebuild costs one HTML entry + one Data Cache entry per tagged `fetch()` on the page).
- **[stocks-crawled-not-indexed-pattern-analysis.md](stocks-crawled-not-indexed-pattern-analysis.md)** — Pattern teardown for the ~95 main `/stocks/{exchange}/{ticker}` URLs that failed the most recent GSC "Crawled — currently not indexed" revalidation. Per-URL teardown of a 48-URL sample (industry, age, visible-word count, sections present), cross-URL patterns (thin content + data-poor inputs on older reports, HTML weight ≫ visible text but driven by RSC streaming payload not charts/CSS, identical OG/Article image on every URL, cron-millisecond `datePublished` clustering [flagged as a *plausible weak signal*, not confirmed], trailing-whitespace industry names, non-US under-linking, `lastmod` granularity), and the data we still need before changing code (full failed-list bucketing + an indexed control group). NASDAQ/CGEN is the key counter-example: 7.6k words *with* Management Team, re-modified 46 days ago, **still failing** — backfilling alone is not sufficient.
- **[aws-migration-plan.md](aws-migration-plan.md)** — Plan to move `insights-ui` off Vercel onto a Terraform-managed AWS deployment on a **single-node Lightsail Container Service**, rolled out in parallel with Vercel: Phase A runs the app directly at **`prod.koalagains.com`** (no CloudFront), static served from **S3** via `assetPrefix`, DB on the **existing public RDS**, **crons moved to EventBridge** (removed from `vercel.json`); Phase B puts CloudFront in front; Phase C retires Vercel. Covers the required app/code changes (`output: 'standalone'`, `assetPrefix`, `VERCEL_ENV` auth coupling, `waitUntil`→`after()`, `maxDuration=300` async refactor, single-node cache coherence), CI/CD, deploy-skew, runbook, rollback, and cost. The Terraform + Dockerfile that implement it live in [`deployments/insights-ui/`](../../deployments/insights-ui/).
- **[aws-deployment.md](aws-deployment.md)** — The **as-built** companion to the migration plan: how the **live** Phase-A AWS deployment actually works now that it's up. Live resource inventory (Lightsail service, ECR, S3 static bucket, Terraform state backend without DynamoDB, Secrets Manager, the `insights-ui-deploy` IAM user + GitHub static-key auth, CloudWatch logs, EventBridge crons), the ordered CI deploy flow, the **env-var / base-URL contract** (the `NEXT_PUBLIC_VERCEL_*`-are-set-on-AWS-not-Vercel asymmetry, and the `getBaseUrlForServerSidePages()` self-fetch vs `getCanonicalUrl()` split that keeps AWS SSR fresh while sitemaps stay canonical), a **table of every real issue hit bringing it up and its fix** (pnpm overrides/build-scripts, lazy Gemini clients for `next build`, Lightsail cert `for_each`, Prisma `debian-openssl-3.0.x` binary target, base-URL staleness), and an operational runbook (redeploy, health/state, container + CloudWatch logs, cert-validation retry). Read this when deploying, debugging a failed AWS deploy, or changing base-URL logic.
- **[cloudfront-error-caching.md](cloudfront-error-caching.md)** — How CloudFront handles **error / not-found** responses and how to stop a transient failure (DB momentarily down, a missing ticker/ETF) from being pinned at the edge for the 6-day TTL. Explains the mechanism (the 6-day policy applies only to 2xx/3xx; 4xx/5xx use the separate Error Caching Minimum TTL), the applied fix (`error_caching_min_ttl=0` for 404/500/502/503/504 in `dodao-api-v2-deployment/cloudfront.tf`, so genuine errors are never cached), and the **residual gap**: `force-dynamic` + Suspense streaming flush a **200** shell before `notFound()`/throw resolves, so soft-error/not-found pages are 2xx and force-cached — fixable only with an app change (await the existence check pre-stream; return real 404 from `full-render` on a miss; stop swallowing 5xx in slice fetchers). Includes a **detection + cleanup runbook** (curl probes, Athena over access logs, CloudWatch error-window, then invalidate via existing tooling).
- **[cloudfront-deploy-skew.md](cloudfront-deploy-skew.md)** — Full caching architecture for `/stocks/*` and `/etfs/*` on koalagains.com: the three-phase journey from `force-static` + ISR (too expensive due to ISR writes flushed on every deploy), through adding CloudFront in front of Vercel (introduced a deploy-skew "Something went wrong" bug), to today's `force-dynamic` + CloudFront + Vercel Skew Protection setup. Covers the deploy-skew fix in detail, where CloudFront is still slow (cold-cache misses, long-tail pages, RSC navigation), the constraints we operate under (no deploy invalidations, zero ISR writes, low function invocations), and the three options for the next iteration (RSC caching via CloudFront Function, selective Cache-Control via Next.js middleware, or status quo).
